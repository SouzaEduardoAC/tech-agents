- # Feature: Support Multi-Client Paths for Gemini, AntiGravity, and Codex
	- **Problem**: The bootstrap tool and the MCP server only supported nested `~/.gemini` folders (e.g. `~/.gemini/antigravity/commands` or `~/.gemini/agents/`), but the actual systems like Codex App/CLI/IDE and AntiGravity App/CLI/IDE store their configurations, command definitions, and personas in their own root-level directories (e.g. `~/.antigravity`, `~/.codex`, etc.). This prevented bootstrap installations and persona resolution from working correctly in Codex and AntiGravity environments.
	- **Solution**:
		- Expanded `CLIENT_SUBDIRS` in `bin/agent-hub.js` to dynamically generate commands, agents, and brain directories for all clients (Gemini, AntiGravity CLI/IDE/App, Codex CLI/IDE/App), including both root-level and nested fallback structures.
		- Configured bootstrap to write `mcp_config.json` and `settings.json` configurations dynamically across all client folders.
		- Upgraded `index.js` probe resolution to recognize and strip persona paths for all client prefix folders (e.g. `~/.antigravity/agents/`, `~/.codex/agents/`, etc.).
		- Made TOML slash command copy-rewriting logic in `bin/agent-hub.js` generic via regex to support rewriting paths for all clients.
	- **Documentation Update**: Synchronized `bin/agent-hub.js`, `index.js`, `docs/pages/registry.md`, `CHANGELOG.md`, and created `docs/journals/2026_07_09.md`. (ref: `bin/agent-hub.js`, `index.js`, `docs/pages/registry.md`, `CHANGELOG.md`)
- # Feature: Optimize Squad Orchestration, Architect Plan, TDD, and Peer Review
	- **Problem**: The squad run command with the architect and developer create commands were not working correctly. Architect Create is used to draft implementation plans but was performing the entire execution/audit protocol. Developers were not writing tests first, and there was no SonarQube code review verification or auto-commits in the pipeline.
	- **Solution**:
		- Updated `squad/commands/squad/run.toml` to invoke `architect:squad-plan`, enforce test-driven development (TDD) for developer agents, and add Phase 4.5 for local peer code reviews with SonarQube MCP integration.
		- Updated developer create commands (`backend`, `frontend`, `mobile`) to skip discovery/architectural planning when in SQUAD-FLOW, and enforce writing tests first before implementing code.
		- Upgraded developer review commands to support **Local Review Mode** when target is local. If clean, the reviewer commits changes to a new `feature/[feature-name]` branch following Conventional Commits, using SonarQube MCP `issues` tool for code scans.
	- **Documentation Update**: Synchronized `index.js`, `squad/commands/squad/run.toml`, developer create/review commands, `docs/pages/Architect.md`, `docs/pages/Squad Orchestrator.md`, and `docs/pages/registry.md`. (ref: `index.js`, `squad/commands/squad/run.toml`, `docs/pages/Architect.md`, `docs/pages/Squad Orchestrator.md`, `docs/pages/registry.md`)

- # Fix: Squad Run ↔ Agent Command Conflict Resolution
	- **Problem**: Cross-analysis of `squad/commands/squad/run.toml` against all `squad-*` agent commands revealed 6 actionable conflicts: (1) the `discovery` gate called by `architect:squad-plan` was never registered in `pipeline_start`, making it a phantom gate; (2) Phase 4 checked `gate="plan"` instead of `gate="compliance"`, allowing compliance to be bypassed; (3) the compliance skip evaluation happened after calling the compliance agent; (4) git commit ownership was inverted — `squad-review` committed but `squad-create` was told not to; (5) Phase 2 checkpoint named one artifact but the Architect produces two; (6) developer phase labels were misaligned between orchestrator (Phase 4) and agents (Phase 3).
	- **Solution**:
		- Registered `"discovery"` gate in `pipeline_start` and added Phase 2a/2b checkpoint blocks in `run.toml`.
		- Fixed Phase 4 gate check: `gate="plan"` → `gate="compliance"`.
		- Restructured Phase 3: compliance skip decision now evaluated before any agent call.
		- Established canonical developer commit cycle: `squad-create` asks human for branch name approval → commits → notifies orchestrator. `squad-review` reviews committed branch → if `APPROVE` pushes to origin; if `REQUEST CHANGES` loops back to developer for fixes and re-commit.
		- Updated Phase 2 checkpoint text to present both `{{args}}-analysis.md` and `{{args}}-architecture.md`.
		- Aligned Phase 4 label across all `squad-create` commands to match orchestrator numbering.
	- **Documentation Update**: Synchronized `squad/commands/squad/run.toml`, `backend/squad-create.toml`, `frontend/squad-create.toml`, `mobile/squad-create.toml`, `backend/squad-review.toml`, `frontend/squad-review.toml`, `mobile/squad-review.toml`, `CHANGELOG.md`, and this journal. (ref: `squad/commands/squad/run.toml`)
- # Fix: Standalone vs Squad Command Alignment (Phase 2)
	- **Problem**: Cross-analysis of all standalone agent commands against their squad counterparts revealed 8 actionable issues: (1) `backend:create` and `frontend:create` were missing a `discovery` gate after Phase 1 — the analysis phase completed with no human checkpoint; (2) the `check_gate("plan")` instruction in both files was orphaned (placed after a `halt`, making it unreachable) — Phase 3 had no guard at entry; (3) all 3 `squad-review` commands were not loading `common/skills/pr_review.md` — the review protocol skill was inlined as a subset, meaning upstream skill updates would not propagate; (4) `compliance:audit` had zero MCP approval gates — the 7-phase audit ran from scope to remediation roadmap without any forced human checkpoint; (5) `compliance:audit` incorrectly loaded `po/knowledge/gatekeeping.md`, a PO-domain skill with no role in a compliance audit (copy-paste artifact); (6) all 6 `review` commands (standalone + squad) lacked cross-references — users discovering one wouldn't know the other existed; (7) standalone `review` commands did not explicitly state they are external-PR-only and do not push; (8) Frontend and Mobile docs pages still referenced Stitch as the sole design tool rather than treating it as one optional option.
	- **Solution**:
		- Added `Gate 0 — MCP Structural Enforcement` (`request_approval(gate="discovery")` + halt) after Phase 1 in `backend:create` and `frontend:create`, consistent with how `architect:create` and `mobile:create` already behave.
		- Moved the orphaned `check_gate("plan")` instruction from Phase 2 (after halt, unreachable) to the opening of Phase 3 in `backend:create` and `frontend:create`.
		- Added `common/skills/pr_review.md` to the skill stack of `backend:squad-review`, `frontend:squad-review`, and `mobile:squad-review` — same review rules, different execution scope.
		- Added `request_approval(gate="compliance")` gate after Phase 6 (Evidence Checklist) in `compliance:audit`, blocking Phase 7 (Remediation Roadmap) until human explicitly acknowledges findings.
		- Removed `po/knowledge/gatekeeping.md` from `compliance:audit` — wrong domain, no role in a compliance audit.
		- Updated descriptions for all 6 `review` commands to cross-reference their counterpart and clarify scope: standalone = external PR, no push; squad-review = local worktree, pushes on APPROVE.
		- Generalized Stitch-specific language in `frontend:create`, `Frontend.md`, and `Mobile.md` to "design tool (e.g. Stitch, Figma, Lovable)" — Stitch is optional, not mandated.
	- **Documentation Update**: Synchronized `Backend.md`, `Frontend.md`, `Mobile.md`, `Compliance.md`, `CHANGELOG.md`, and this journal. (ref: `backend/commands/backend/create.toml`, `frontend/commands/frontend/create.toml`, `compliance/commands/compliance/audit.toml`, `docs/pages/Backend.md`, `docs/pages/Frontend.md`, `docs/pages/Mobile.md`, `docs/pages/Compliance.md`)
- # Feature: Upgrade Squad & Developer Run Completeness Criteria
	- **Problem**: In agent-hub developer runs, developer agents would sometimes declare the implementation phase complete early, skipping required steps like updating documentation, proposing the branch name, getting user confirmation, creating the commit, and pushing the branch to origin.
	- **Solution**:
		- Upgraded the developer agents' `squad-create` commands (`backend`, `frontend`, `mobile`) to ask for branch name, commit, push the branch to origin, and notify.
		- Injected a critical warning enforcing that the developer agents treat "implementation complete" as incomplete until all 5 steps are fulfilled.
		- Upgraded `squad/commands/squad/run.toml` to reflect the updated developer flow (including branch push) and to enforce the same 5-step completeness check.
	- **Documentation Update**: Synchronized `squad/commands/squad/run.toml`, `backend/squad-create.toml`, `frontend/squad-create.toml`, `mobile/squad-create.toml`, `docs/pages/Squad Orchestrator.md`, `docs/pages/Backend.md`, `docs/pages/Frontend.md`, `docs/pages/Mobile.md`, `docs/pages/registry.md`, and this journal. - # Feature: Mandatory Commit Scope & Body Standard
	- **Problem**: Conventional commits made by agents were sometimes too brief, omitting the scope (making it hard to track which module was modified) or the body (omitting the explanation of *why* the change was made).
	- **Solution**:
		- Upgraded `common/knowledge/git_standard.md` to mandate the type(scope): description format and require a body for all commits.
		- Upgraded `common/skills/base_reviewer.md` and `common/skills/pr_review.md` to verify that commit messages and PR titles strictly match the updated standard.
	- **Documentation Update**: Synchronized `common/knowledge/git_standard.md`, `common/skills/base_reviewer.md`, `common/skills/pr_review.md`, `docs/pages/TECHNICAL_SPECS.md`, `docs/pages/registry.md`, and this journal.

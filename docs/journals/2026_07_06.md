- # Feature: Upgrade Squad Orchestrator delegation protocol
	- **Problem**: In the unified orchestrator command `squad/commands/squad/run.toml`, child agents (PO, Architect, Compliance, Developers, and Decoder) were called using `get_agent_prompt(agent="...")` instead of `call_agent_command(agent="...", command="...", args="...")`. This only loaded their base personas, skills, and knowledge files, bypassing their custom command TOML prompts (such as `discovery.toml` or `create.toml`). Consequently, step-by-step instructions (e.g. design extraction via Stitch, visual verification via Playwright, analysis and audit generation) were not loaded.
	- **Upgrade**: Replaced all five instances of `get_agent_prompt` with `call_agent_command` with their corresponding command entrypoints (`discovery`, `create`, `master`, and `export`) to fully activate the specialized agent protocols.
	- **Documentation Update**: Updated `docs/pages/Squad Orchestrator.md` and added an entry to `CHANGELOG.md`. (ref: `squad/commands/squad/run.toml`, `docs/pages/Squad Orchestrator.md`, `CHANGELOG.md`)

- # Feature: Remove Automata from Squad Execution
	- **Raison d'être**: The Automata agent (which architects and generates JSON files for workflow automation platform imports) does not belong inside the primary Squad Orchestrator's code execution phase (Phase 4). Its role is automation, not software code creation (which is handled by Backend, Frontend, and Mobile).
	- **Changes**: Removed all references to the Automata agent from Phase 4 routing in the Squad persona and TOML command scripts.
	- **Documentation Update**: Synchronized `squad/brain/persona.md`, `squad/commands/squad/run.toml`, `docs/pages/Squad Orchestrator.md`, `docs/pages/registry.md`, and `CHANGELOG.md`. (ref: `squad/brain/persona.md`, `squad/commands/squad/run.toml`, `docs/pages/Squad Orchestrator.md`, `docs/pages/registry.md`, `CHANGELOG.md`)

- # Feature: Isolate Automata as Standalone Agent
	- **Raison d'être**: Following its removal from the Squad's direct code execution lifecycle, Automata has been decoupled from the immediate "Extended Team" context of the Product Owner and Systems Architect personas. This isolates it as a standalone specialist agent for workflow automation, accessible only when explicitly triggered by the user or when recommended as an external integration checkpoint.
	- **Changes**: Removed Automata from the `Extended Team` delegation lists in `po/brain/persona.md` and `architect/brain/persona.md`.
	- **Documentation Update**: Synchronized `po/brain/persona.md`, `architect/brain/persona.md`, `docs/pages/registry.md`, and `CHANGELOG.md`. (ref: `po/brain/persona.md`, `architect/brain/persona.md`, `docs/pages/registry.md`, `CHANGELOG.md`)

- # Feature: Branch-Scoped Pipeline State Files
	- **Problem**: Previously, the pipeline state file was named statically as `.squad-state.json` at the project root. This caused session collisions and deadlocks when: (1) multiple developers or clients ran different pipelines concurrently in the same project workspace, or (2) when identical branch names (e.g. `migrate-to-dotnet-10`) were checked out in separate projects but connected to a global MCP daemon process (which locks CWD to `~/` and cannot resolve the project path).
	- **Solution**: Upgraded `index.js -> resolveStateFilePath()` to dynamically query the active git branch name, sanitizing it to suffix the state file as `.squad-state-[branch].json`. Extended the tool schemas for `pipeline_start`, `request_approval`, `check_gate`, and `pipeline_approve` to accept an optional `cwd` parameter. If provided, the server resolves the project root relative to that path instead of relying strictly on `process.cwd()`, fully isolating identical branch names across separate project workspaces under global daemons. Adjusted `.gitignore` logic to append `.squad-state-*.json`. Refactored `squad/commands/squad/approve.toml` to delegate to the MCP `pipeline_approve` tool.
	- **Documentation Update**: Synchronized `index.js`, `squad/commands/squad/approve.toml`, and `CHANGELOG.md`. (ref: `index.js`, `squad/commands/squad/approve.toml`, `CHANGELOG.md`)

- # Feature: Remote Unversioned Agent Hub MCP Configuration
	- **Problem**: The MCP configurations for various LLMs (AntiGravity CLI, IDE, and 2.0) were pointing to specific versioned tags (`github:SouzaEduardoAC/tech-agents#v1.13.0`), which caused them to run outdated logic or fail to utilize latest feature upgrades unless manual configuration updates were made.
	- **Solution**: Modified all local and global MCP configuration files (such as `mcp_config.json` and `settings.json`) to point directly to the remote URL (`https://github.com/SouzaEduardoAC/tech-agents`) without version constraints, adding the `-y` and `--prefer-online` flags to force `npx` to fetch the latest version from origin on execution. Updated the local `bin/agent-hub.js` bootstrap script and Claude desktop configuration templates to align with this configuration.
	- **Documentation Update**: Updated `docs/journals/2026_07_06.md`, `docs/pages/registry.md`, and `CHANGELOG.md`. (ref: `bin/agent-hub.js`, `docs/pages/registry.md`, `CHANGELOG.md`)

- # Fix: Squad `run.toml` Prompt-Level Human Approval Guardrails
	- **Problem**: The Squad `run.toml` orchestration prompt did not enforce human approval gates at the instruction level, allowing LLM orchestrators (e.g., Claude Code) to auto-approve phase transitions and bypass the Quality Control mandate in `squad/brain/persona.md`.
	- **Three Checkpoint Blocks Added**: Inserted explicit `âš ď¸Ź MANDATORY HUMAN CHECKPOINT â€” PIPELINE PAUSED` blocks after Phase 1 (PRD), Phase 2 (Implementation Plan), and Phase 3 (Compliance Audit). Each block uses a triple-prohibition pattern (`Do NOT auto-approve. Do NOT proceed autonomously. Do NOT call [next agent].`) and a double-fence gate condition at the start of each subsequent phase.
	- **Documentation Sync**: Registered the fix in `docs/pages/registry.md` (Bug Fix entry) and `CHANGELOG.md` (v1.9.6). (ref: `squad/commands/squad/run.toml`, `docs/pages/registry.md`)

- # Feature: MCP-Level Structural Human Approval Gate System
	- **Problem**: Prompt-level guardrails are behavioral â€” a sufficiently autonomous LLM can rationalize past them. The fix needed to be structural: make it physically impossible at the API level to call the next pipeline phase's agent without a registered human approval token.
	- **Three New MCP Tools**: Added `pipeline_start`, `request_approval`, and `check_gate` to `index.js`. `pipeline_start` initializes a session state file (`.squad-state.json`) with all gates set to `locked`. `request_approval` advances a gate to `pending` and returns a hard STOP message. `check_gate` returns `isError: true` (a proper MCP error, not a text hint) if a gate is `pending` or `locked`, structurally blocking LLM tool call continuation. (ref: `index.js`)
	- **Human Trust Anchor**: Created `/squad:approve <gate>` command (`squad/commands/squad/approve.toml`) â€” the sole mechanism that can write `approved` status to the state file. LLMs cannot invoke it as an MCP tool; only the human can type it explicitly.
	- **Standalone Mode Grace**: `check_gate` returns a soft advisory (`approved: true`) when no `.squad-state.json` exists, so individual agents (architect, backend, etc.) are not blocked when run outside of a Squad pipeline session.
	- **Pipeline Integration**: `run.toml` updated to call `pipeline_start` at session start and `request_approval`/`check_gate` at all three gate boundaries. (ref: `squad/commands/squad/run.toml`, `squad/commands/squad/approve.toml`)
	- **Agent TOMLs Updated**: All 8 individual agent command TOMLs (architect, backend, frontend, mobile, automata, researcherĂ—2, po) updated with `check_gate`/`request_approval` calls at each phase entry/exit point.
